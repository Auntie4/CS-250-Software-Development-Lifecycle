# CS-250-Software-Development-Lifecycle
# How do I interpret user needs and implement them into a program? How does creating “user stories” help with this?
When I interpret user needs, I begin with proactive data gathering. I analyze user workflows, gather feedback, conduct interviews, and study error reports to map out essential system inputs and outputs. Once gathered, I prioritize these requirements using methods. User stories are a foundational Agile tool for capturing requirements. As a [Type of User], I want to [Take an Action], so that I can [Achieve a Goal]. They translate high-level needs into bite-sized, non-technical features from the user's perspective, ensuring development stays focused on specific goals. Rather than rigid documentation, they serve as essential placeholders for ongoing stakeholder discussions.
# How do I approach developing programs? What Agile processes do I hope to incorporate into my future development work?
I recognize two primary software development methods, Waterfall and Agile. Waterfall follows a direct, solid path where each phase ends before the next starts. Agile, my preferred approach, is iterative and incremental, prioritizing flexibility, continuous testing, and client integration. Looking ahead, I plan to adopt practices like Sprint Planning and Backlog Refinement to collaboratively review user stories, define precise daily tasks, and estimate effort. I also intend to implement daily stand-ups to discuss current goals, previous progress, and any blockers or challenges. Finally, utilizing visual boards will be essential for managing my workflow and identifying potential problem areas or bottlenecks.
# What does it mean to be a good team member in software development?
Teamwork is about more than just writing clean code; it’s really about being accountable to each other. I like to think of us as a unit, if one of us stumbles, we all do, so I’m always ready to help out with bugs and keep our momentum going during sprints. During our planning sessions, I would make it a point to ask questions to make sure everyone is on the same page about what we’re building, which helps us avoid any missteps. I appreciate different perspectives and take feedback from code reviews with an open mind, seeing mistakes as chances for all of us to learn. Plus, I want to always make sure that any code I touch is well-documented, peer-reviewed, tested, and polished before we wrap things up.
